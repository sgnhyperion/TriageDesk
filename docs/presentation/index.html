02 The problem P3

One bot is fast but dangerous. Humans are safe but slow.

AI alone

Fast — but hallucinates answers, promises refunds it shouldn't, and leaks personal data.

Humans alone

Safe — but slow and expensive. Every ticket waits on a person.

TriageDesk

AI resolves in seconds; a human approves anything touching money or the customer.

Users: Tier-1 support agents & admins. Real support is wildly varied — billing, bugs, how-tos, vague asks, out-of-scope.

03 Why it needs multiple agents P1

01

Different jobs → different agents

Understanding a ticket, finding the answer, writing the reply, and checking it for safety are different skills with different failure modes. So we split them into specialized agents.

02

A dynamic brain, not a fixed script

~10 tools; every ticket needs a different combination in a different order. You only learn the next step after the last — look up an order, oh it's a duplicate, now propose a refund. A flowchart would enumerate every path — intractable. A supervisor brain picks tools at runtime, governed by guardrails.

04 Architecture · four agents, one state P1

Four agents. One shared state.

classify

Triage

Category, severity, sentiment, in-scope, confidence.

the brain

Supervisor

Reads state → emits one Decision (tool + why) → loops.

draft

Resolution

Writes a grounded customer reply.

check

QA

PII, policy, hallucination — before any human sees it.

SupportState · scratchpad

One object flows through every step. The scratchpad is the brain's working memory — the ordered history of ToolResults it reads before each decision. That's how it "remembers" the duplicate it just found.

structured handoffs

Every handoff is a Pydantic model — Decision, ToolResult, Classification. Agents fit together by contract, and an invalid tool name is rejected at parse time.

05 The graph · human-in-the-loop P1

START → supervisor ─┬─(continue)──→ tools ──┐ │ │ │ ┌───────────────┘ (loop) │ ▼ ├─(await_human)→ high_impact ◀── interrupt() · PAUSE │ │ │ ├─(approved)→ supervisor │ └─(sent/reject)→ END │ └─(done/escalate/refuse)→ END
How the pause survives

interrupt() persists the whole graph state via a Postgres checkpointer, keyed by thread_id = ticket_id.

  • Approval can arrive in a separate HTTP request
  • It survives a server restart
  • Command(resume=…) reloads and continues the exact same run

06 Tools · data · RAG P2

10 tools. Real Postgres. Grounded answers.

10 tools → ToolResult

CRM lookups, past-ticket search, bug filing, refund, draft, email — every one returns a structured result, backed by real Postgres.

RAG over pgvector

Help docs → chunk → embed to 768-dim vectors → search by meaning. A grounding threshold: below it, escalate — don't hallucinate.

Mocked on purpose

Inbound email (a "new ticket" form) and the refund (writes a row, no Stripe). The approval flow is identical either way.

handoffs all Pydantic — Decision · ToolResult · Classification

lookup_customer→ lookup_order→ draft_reply→ retrieve_kb→ send_email

09 Guardrails · the governance P1

Eight deterministic guardrails around the LLM.

Out-of-scope→ refuse
Low confidence→ escalate
Critical severity→ escalate
Step budget = 8→ escalate · no infinite loops
Grounding floor→ escalate · no hallucination
QA / PII pass→ on every draft
Allow-list→ only a valid tool name
Human approval→ before refund & email

10 Limitations · being honest we know

  • Refund & inbound email are mocked — the approval flow is real either way.
  • The resume path has one reconciliation TODO.
  • RAG is single-vector — no re-ranker yet.
  • The trace is shown after the run, not streamed live.
